# 项目终止复盘与经验沉淀

## 1. 结论先行
- 本项目在“病鱼小目标检测+分割”场景下，经过多轮训练、架构升级和 LLM 自动调参后，未稳定超过基线。
- 复盘结论不是“只差一点调参”，而是“数据规模、架构改动幅度、预训练继承效率、训练稳定性”同时构成瓶颈。
- 可沉淀的最大价值是方法论：哪些尝试有效、哪些路径高风险、如何构建下一代自动调参系统。

## 2. 问题画像
- 任务难度高: 目标小、背景噪声重、样本分布复杂。
- 训练环境复杂: Windows + 新显卡架构导致兼容性与稳定性问题频发。
- 多任务优化难: 检测与分割目标存在梯度冲突，收敛容易不稳定。
- 架构改动过大: 预训练权重匹配比例下降，导致“改完像新模型，从头学”。

## 3. 已做尝试（按层次）
### 3.1 架构层
- 引入 P2 小目标检测头。
- 引入 ContextAggregation、CBAM 等模块强化特征表达。
- 检测头输入从多尺度进一步扩展（提升小目标召回预期）。

### 3.2 损失与优化层
- 尝试引入 NWD Loss 改善小目标定位鲁棒性。
- 使用 MTL 不确定性加权，尝试 PCGrad/CAGrad 缓解梯度冲突。
- 对数值不稳定部分加入 clamp 等保护机制，减少 NaN 风险。

### 3.3 调参与执行层
- 多轮 LLM 自动调参（学习率、增强策略、训练策略组合搜索）。
- 多轮人工调参兜底（冻结策略、学习率、增强幅度、分辨率等）。
- DataLoader/线程/内存参数联动修复，优先保证训练可连续执行。

## 4. 关键失败机理（可复用）
- 失败机理 A: 架构改动过大 + 数据有限 -> 预训练红利流失，训练变成“低数据从零学习”。
- 失败机理 B: 自动调参在高噪声条件下容易“局部最优循环”，看似探索充分，实则重复。
- 失败机理 C: 工程补丁虽然保住可运行，但吞吐下降和训练噪声增加，会压制搜索效率。
- 失败机理 D: 未先建立严格的“实验协议与停机条件”，导致计算投入与信息增益不匹配。

## 5. 可迁移经验
- 先控变量，再上 LLM: 基线、数据、环境三者未锁定前，不应大规模自动调参。
- 先保预训练继承，再谈结构创新: 若权重匹配比例过低，优先降低结构改动幅度。
- 先做小规模判别实验，再全量训练: 低成本验证假设，避免“150~500 epoch 盲跑”。
- 为 LLM 设边界: 明确可调参数集合、步长、约束和拒绝条件，减少发散。
- 把失败也标准化记录: 失败实验应成为下一轮策略检索的高价值样本。

## 6. 下一代项目建议（抽象）
- 以“实验协议”驱动而非“脚本驱动”:
  - 固定随机种子、固定评估集、固定日志字段。
  - 每轮实验必须带假设、预期、停止条件。
- 构建“检索增强的调参系统”:
  - LLM 不直接凭直觉给参数，而是先检索历史实验模式再给方案。
- 明确终止标准:
  - 连续 N 轮无统计显著提升则自动触发策略降级或停止。
