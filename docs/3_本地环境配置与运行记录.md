# 3. 本地环境配置与运行记录

## 1. 依赖分析与安装

在项目根目录下存在 `requirements.txt`，其中包括 `torch`、`torchvision`、`opencv-python` 等常见深度学习与计算机视觉依赖。

### 1.1 `requirements.txt` 修改
原始的 `requirements.txt` 中包含了一些过于陈旧且不兼容当前 Python 版本的依赖限制，例如：
- `torchvision==0.8.0`：版本过低，导致找不到对应的高版本 PyTorch 匹配。
- `protobuf<4.21.3`：存在潜在兼容性问题。

**解决思路**：
编辑了 `requirements.txt`，去除了 `torchvision` 和 `protobuf` 的强制旧版本限制，允许 pip 安装兼容当前环境的较新版本。

### 1.2 运行依赖安装
执行了如下命令：
```bash
pip install -r requirements.txt
```
所有基本依赖成功安装。

## 2. 预测脚本 (`predict.py`) 测试与报错排查

在执行预训练模型预测测试时，遇到了多个缺失依赖和源码兼容性问题。

### 2.1 缺失依赖 `mmengine` 和 `mmcv`

**问题现象**：
运行 `python predict.py` 时报错 `ModuleNotFoundError: No module named 'mmengine'`，安装 `mmengine` 后再次运行报错 `ModuleNotFoundError: No module named 'mmcv'`。

**解决思路**：
`mmcv` 的安装在当前 Windows 环境下可能因为编译环境和库版本（特别是 `setuptools` 的兼容性问题）经常出现困难。
经排查源码发现，仅在 `ultralytics/nn/modules/Ctxt.py` 中引入了 `mmcv.cnn.ConvModule`：
```python
from mmcv.cnn import ConvModule
```
而该类在这里的使用场景非常简单，只是作为一个基础卷积层。为了避免繁琐的 `mmcv` 编译安装，我直接在 `Ctxt.py` 中自定义了一个轻量级的 `ConvModule`：
```python
import torch.nn as nn
class ConvModule(nn.Module):
    def __init__(self, in_channels, out_channels, kernel_size, conv_cfg=None, act_cfg=None):
        super().__init__()
        self.conv = nn.Conv2d(in_channels, out_channels, kernel_size)
    def forward(self, x):
        return self.conv(x)
```
这一修改成功移除了项目对 `mmcv` 库的强制依赖。

### 2.2 缺失依赖 `einops`

**问题现象**：
再次运行预测时，在加载 `Biformer` 模型模块时报错 `ModuleNotFoundError: No module named 'einops'`。

**解决思路**：
直接执行 `pip install einops` 进行安装，成功解决。

### 2.3 PyTorch 2.6 安全加载机制导致模型加载失败

**问题现象**：
运行报错：`_pickle.UnpicklingError: Weights only load failed. ...` 
原因在于 PyTorch 2.6 版本后 `torch.load` 的 `weights_only` 参数默认变为 `True`，导致包含自定义对象（如 `ultralytics.nn.tasks.DetectionModel`）的 `.pt` 模型文件无法直接加载。

**解决思路**：
修改 `ultralytics/nn/tasks.py` 中的 `torch_safe_load` 函数，将加载语句显式修改为允许反序列化：
```python
return torch.load(file, map_location='cpu', weights_only=False), file
```

### 2.4 CPU 环境下代码强制使用 CUDA 的问题

**问题现象**：
预测视频时抛出异常：`AssertionError: Torch not compiled with CUDA enabled`。
排查发现 `ultralytics/yolo/utils/ops.py` 中存在硬编码将张量分配到 GPU 的操作，例如 `.to('cuda')` 以及 `device='cuda'`。

**解决思路**：
修改 `ultralytics/yolo/utils/ops.py` 相关的设备分配代码，将其改为动态跟随输入张量的设备：
1. `scale_seg_mask_cuda` 中：
   - 将 `pad = torch.tensor(pad).to('cuda')` 修改为 `pad = torch.tensor(pad).to(mask.device)`。
2. `colorEncode_cuda` 中：
   - 将 `colors = torch.tensor(colors).to('cuda')`（或类似硬编码）修改为 `colors = torch.tensor(colors).to(labelmap.device)`。
   - 将 `torch.zeros(..., device='cuda')` 修改为 `torch.zeros(..., device=labelmap.device)`。
