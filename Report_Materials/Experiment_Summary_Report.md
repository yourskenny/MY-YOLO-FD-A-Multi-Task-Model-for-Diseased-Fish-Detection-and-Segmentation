## 1. 我尝试修改的网络架构与损失函数

这段时间因为我们数据集里的鱼类病灶都非常小，且水下图像背景干扰大，所以我查阅了一些资料，自作主张对原本的 YOLO 网络做了一些“魔改”：
- **P2 小目标检测头**：我给网络增加了一个更浅层的高分辨率检测头，希望能抓到更微小的病灶。
- **ContextAggregation 与 CBAM**：我在网络里插入了多尺度上下文聚合模块和注意力机制，目的是想抑制水波纹等背景噪声。
- **NWD Loss (Normalized Gaussian Wasserstein Distance)**：传统的 IoU 对小目标太敏感了，差几个像素 IoU 就掉光了，所以我尝试换成了 NWD Loss。

**遇到的坑与困惑：** 
改完之后网络变得非常臃肿。NWD Loss 刚加上去的时候，因为张量维度（Batch Size 广播）没对齐，报了一堆 `The shape of the mask does not match...` 的错。虽然最后用 `view(-1)` 强行拉平维度把代码跑通了，但我其实不太确定这种修改会不会破坏底层的梯度反传逻辑。

## 2. 跑代码时遇到的工程 Bug（耗费了大量时间）

由于我是第一次在本地 Windows 环境和 RTX 5070 Ti（新架构）上跑这么大的多任务网络，我遇到了非常多底层的 Bug，大半的时间都在跟环境做斗争：
1. **CUDA 兼容性问题**：一开始总是报 `no kernel image is available`，后来查了半天才发现官方稳定版 PyTorch 不支持 50 系显卡的 sm_120 架构，只能换成 Nightly 版的 CUDA 12.8 环境才跑起来。
2. **多进程卡死（DataLoader Deadlock）**：只要我把 `workers` 设置大于 0，训练就死死卡在 0% 进度条不动。后来发现是 Windows 多进程和 PyTorch 的内存锁页有冲突。我只能把 `os.environ['PIN_MEMORY']` 和底层源码里的 `pin_memory` 全部强行改成 `False`，并关掉 OpenCV 的多线程才勉强跑通。
3. **多任务（MTL）梯度爆炸**：我们的网络要同时算检测 Loss 和分割 Loss，我发现好几次跑着跑着 Loss 就变成了 NaN。最后我只能在代码里加了 `torch.clamp`，强行限制权重的上下界。

**求助点**：虽然这些 Bug 表面上被我用“打补丁”的方式修好了，但我很担心这种强行 `clamp` 截断和关掉 `pin_memory` 的操作，是不是实际上已经损害了模型的训练上限和收敛速度？

## 3. 乱调超参数的教训（导致跑分断崖式下跌）

因为自己不知道该怎么调参，我一开始甚至写了个脚本让大语言模型（LLM）来盲调，跑了几十轮，最高也就到了 0.751，还是没超过基准。后来我尝试自己当“专家”手动定了一组策略，结果闹了笑话：

- **错误冻结导致的灾难 (Train84)**：我以为要继承初始权重，就在配置里写了 `freeze=10`，并给了极低的学习率 `0.001`。结果跑分直接暴跌到 `0.666`。
- **我后知后觉的原因**：我看日志才发现，因为我对网络魔改得太厉害了，原本 670 层的基准权重，**实际只加载成功了 53 层！** 我把这 53 层（Backbone）冻结了，导致后面几百层全新初始化的随机网络（包括 P2 头）根本得不到有效训练。
- **解冻后的回升 (Train85)**：发现这个低级错误后，我赶紧把 `freeze` 改回 0，把学习率拉高到 `0.005`，并降低了 MixUp 这种会破坏小目标的数据增强。跑分终于回到了 `0.743`，但依然没能突破接手时的 `0.760`。
